Critical/Code/Studies

Source code is not only a set of instructions to a machine. It is writing — authored, situated, ideological, readable. We read the programs that made computing what it is, line by line, as cultural texts.

What we do

The premise: code can be read

Critical Code Studies (CCS) applies the interpretive practices of the humanities to the extra-functional meaning of computer source code. A program does its work, but it also carries assumptions, histories, metaphors, and politics in its variable names, its comments, its data structures, its silences and its defaults. CCS reads those.

This site gathers a series of sustained close readings of landmark programs, together with the method that holds them together. Each reading takes a single historic system and reads its actual source — not a summary, not a screenshot, but the code — as a document of its moment.

The readings

Programs read as cultural texts

Three landmark systems from the first decade of interactive computing, each read in depth. More are in preparation.

MIT · 1968–70 SHRDLU

Terry Winograd's natural-language program for the blocks world. Reading the code at the high-water mark of symbolic AI, in MacLisp and Micro-Planner.

Live · read now MIT · 1962 Spacewar!

Steve Russell and the TMRC hackers on the PDP-1. The first widely shared video game, read through its assembly source and its vector-display physics.

In preparation MIT · 1966 ELIZA

Joseph Weizenbaum's DOCTOR script and the SLIP language beneath it. The founding CCS reading — the conversational illusion taken apart line by line.

How we read

Close reading, for code

CCS does not treat code as a black box to be benchmarked, nor as mere documentation of a system's behaviour. It treats the source as a text with an inside. The method borrows from literary criticism, science and technology studies, and the history of computing, and it asks a consistent set of questions of every program.

01

Read the source itself

Work from the actual code — the listing, the comments, the identifiers — not a paraphrase of what the program does.

02

Mind the defaults

Where a program falls through, hard-codes, or stays silent is where its assumptions live. The default branch is rarely neutral.

03

Situate the system

A program is written in a language, on a machine, in an institution, at a moment. The material conditions are part of the meaning.

04

Follow the metaphors

"Blocks world", "the doctor", "the player": naming is theory. The figures a program reaches for shape what it can and cannot imagine.

Code is not the law of the machine alone; it is a cultural and material artefact, and it can be read as one. The CCS premise

Who reads

These readings are produced as a collaboration, co-led by David M. Berry (University of Sussex) and Mark C. Marino (University of Southern California), drawing on the wider Critical Code Studies community that has met since 2010 in the CCS Working Group.
