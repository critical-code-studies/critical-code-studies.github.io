Method

Reading code as a cultural text

Critical Code Studies is the application of critical hermeneutics to the interpretation of computer source code, program architecture, and documentation within a sociohistorical context. It holds that code is not exhausted by what it computes. Like any writing, code means more than it says, and it can be read for that surplus.

The premise

Code is writing

Source code occupies an unusual position. It is functional — it must run, and it answers to the machine — and it is also a text addressed to other people: collaborators, maintainers, students, future readers. It is written in named languages, with conventions, idioms, and styles. It carries comments, which are pure address to a human reader. It chooses metaphors. It is authored under conditions, by people, in institutions, for purposes.

That double character is what makes code available to interpretation. A variable named patient rather than user encodes a theory of the encounter. A hard-coded default encodes an expectation about who the program is for. A comment that apologises for a hack records a history of constraint. None of this changes what the program computes, and all of it is meaning.

We do not read code to find out what it does. We read it to find out what it assumes. Critical Code Studies
How we read

Four moves

Every reading on this site works through the same set of questions. They are not a checklist so much as a discipline — a way of staying with the source rather than retreating to a summary of its behaviour.

01

Read the source itself

The unit of analysis is the actual code: the listing, the identifiers, the comments, the data structures. Not a description of what the program does, but the document that does it. Where the original source survives, we read it; where it does not, we reconstruct from the best available record and say so.

02

Mind the defaults and silences

The most revealing lines are often the ones that handle the unhandled case: the fall-through, the catch-all, the magic constant, the thing left unwritten. A program's assumptions live where it stops making distinctions.

03

Situate the system materially

A program is written in a particular language, for a particular machine, under a particular operating system, inside a particular institution, at a particular moment. MacLisp on the ITS, assembly on the PDP-1: the medium is part of the meaning, not a backdrop to it.

04

Follow the metaphors

"Blocks world", "the doctor", "the player", "the kill". Naming is theory under another name. The figures a program reaches for set the horizon of what it can imagine, and tracing them is one of the surest routes into its politics.

Lineage

Where this comes from

The term Critical Code Studies was proposed by Mark C. Marino in 2006, and the field has developed since 2010 through the online Critical Code Studies Working Group and the biennial CCS conferences. Its touchstones include Marino's Critical Code Studies (MIT Press, 2020), the collaboratively authored 10 PRINT CHR$(205.5+RND(1)); : GOTO 10 (2013), and the wider currents of software studies, platform studies, and the digital humanities.

The readings collected here continue that work, with an emphasis on sustained, single-program close readings presented as standalone, citable sites. See the bibliography for the foundational literature, and the readings for the programs themselves.
